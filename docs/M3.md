## 3. 요구사항 정의

### 기능 요구사항 (Functional Requirements)

| ID | 요구사항 내용 | 우선순위 |
|----|--------------|---------|
| FR-01 | 시스템은 사용자가 운동 수행 중 자신의 자세를 실시간으로 확인할 수 있어야 한다 | 상 |
| FR-02 | 시스템은 사용자의 자세 오류를 감지하고 즉각적으로 피드백을 제공할 수 있어야 한다 | 상 |
| FR-03 | 시스템은 관절에 부담이 되는 동작이 발생할 경우 사용자에게 경고를 제공할 수 있어야 한다 | 상 |
| FR-04 | 시스템은 헬스장 환경에 따라 대체 가능한 운동을 사용자에게 제안할 수 있어야 한다 | 중 |
| FR-05 | 시스템은 사용자의 컨디션에 따라 운동 강도를 조정하여 추천할 수 있어야 한다 | 중 |
| FR-06 | 시스템은 사용자가 업로드한 음식 사진을 분석하여 칼로리 및 영양소 정보를 제공할 수 있어야 한다 | 중 |
| FR-07 | 시스템은 사용자의 운동 및 식단 데이터를 기반으로 주간·월간 성과 리포트를 제공할 수 있어야 한다 | 중 |
| FR-08 | 시스템은 사용자가 이메일과 비밀번호로 회원가입할 수 있어야 한다 | 상 |
| FR-09 | 시스템은 사용자가 등록된 계정으로 로그인할 수 있어야 한다 | 상 |
| FR-10 | 시스템은 사용자가 피로도 및 운동 목표를 입력할 수 있어야 한다 | 중 |
| FR-11 | 시스템은 사용자가 기능 피드백을 제출할 수 있어야 한다 | 하 |
| FR-12 | 시스템은 사용자가 자신이 제출한 피드백 목록을 조회할 수 있어야 한다 | 하 |
| FR-13 | 시스템은 관리자가 피드백을 검토하고 답변을 등록할 수 있어야 한다 | 하 |

## 9. 코딩 표준

본 프로젝트는 향후 실제 구현 단계에서 아래 코딩 표준을 적용할 것을 정의한다.

### 네이밍 규칙

| 대상 | 규칙 | 예시 |
|------|------|------|
| 클래스 | PascalCase | `PostureAnalyzer`, `WorkoutRecommender` |
| 메서드 | camelCase | `analyzeFrame()`, `detectInjuryRisk()` |
| 변수 | camelCase | `fatigueLevel`, `jointData` |
| 상수 | UPPER_SNAKE_CASE | `MAX_FPS`, `ANALYSIS_TIMEOUT` |
| 파일명 | PascalCase (클래스 파일) | `PostureAnalyzer.kt` |

### 코드 품질 기준

- 함수 길이: 단일 함수 50줄 이하 권장
- 주석: 공개 메서드(public method)에 KDoc/JavaDoc 형식 주석 필수
- 중복 코드: 동일 로직 3회 이상 반복 시 함수 추출 원칙 적용
- 에러 처리: 외부 API 호출 및 카메라 접근 시 예외 처리 필수

### 브랜치 전략

```
main          ← 최종 배포 브랜치
develop       ← 통합 개발 브랜치
feature/[기능명] ← 기능 단위 개발 브랜치
fix/[버그명]     ← 버그 수정 브랜치
```

### 커밋 메시지 규칙

```
feat: 새 기능 추가
fix: 버그 수정
docs: 문서 수정
refactor: 코드 리팩토링
test: 테스트 추가
```

## 10. AI 활용 내역

### 사용 도구

| 도구 | 주 활용 용도 |
|------|------------|
| Claude | 요구사항 초안 생성, UML 다이어그램 초안, 모듈 설계서 작성, UI 프로토타입 생성 |
| ChatGPT | 코드 검토 보조, 설계 패턴 조사, 보완적 의견 수집 |

### AI 활용 3원칙

1. **단순 복사 금지**: AI 생성 결과물을 그대로 제출하지 않고 팀 전원 검토 후 수정·보완
2. **비판적 검증**: AI 답변의 논리적·보안적 결함을 반드시 확인
3. **수정 이력 명시**: 프롬프트 기록과 수정/보완 과정을 개인 AI 로그에 구체적으로 작성

### 단계별 AI 활용 내역

## 11. 프로젝트 회고

### 잘된 점

- **역할 분담 명확화**: PM·분석가·설계자·개발자·QA 역할을 초기에 명확히 정의하여 혼선 없이 진행
- **GitHub 활용**: Issues, Labels, Milestones를 통해 작업 진척을 체계적으로 관리
- **문서화 일관성**: FR/NFR ID를 요구사항 정의부터 UML, 모듈 설계까지 일관되게 추적
- **AI 도구 활용**: Claude·ChatGPT를 초안 생성 도구로 활용하면서 결과물을 팀이 직접 검증하는 방식으로 생산성과 품질을 동시에 확보

### 어려웠던 점

- **UML 초안 오류**: AI가 생성한 유스케이스 다이어그램에서 Actor 연결 오류와 관계 방향 오류가 발생하여 인스펙션 단계에서 수정 필요
- **일정 압박**: 설계·구현 단계가 후반부에 집중되어 일부 산출물의 완성도를 높이는 데 시간이 부족
- **원격 협업 한계**: 비대면 환경에서 실시간 의견 조율이 어려워 KakaoTalk 활용이 증가

### 개선 방향

- 다음 프로젝트에서는 설계 단계의 일정을 더 일찍 시작하여 구현 단계에 충분한 시간 확보
- 인스펙션을 단계별로 조기에 실시하여 후반 수정 부담 감소
- AI 생성 결과물에 대한 팀 검토 프로세스를 더 체계화하여 오류 조기 발견
